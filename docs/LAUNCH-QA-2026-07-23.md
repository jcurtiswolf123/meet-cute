## Executive finding

The public site, member app, operator studio, access controls, responsive layouts, build, container, and baseline security checks are in much better launch shape on the QA branch. The branch is not ready for public launch because four operational failure modes remain:

1. Photo storage is local to each Fly machine, so an uploaded photo can disappear when a later request reaches the other machine.
2. Introduction email and SMS delivery does not use a durable delivery queue with provider identifiers, retries, and visible per-channel state.
3. Concierge and booking actions can imply work that no production worker, calendar integration, or reservation system performs.
4. Production still has demo-login secret names and a test-named operator that should be removed or audited before launch.

These are product integrity issues. They can expose inconsistent private media, lose introductions, or tell members that work happened when it did not.

## Fixed on the QA branch

### Access and privacy

- Applicant accounts are redirected away from member routes.
- Approved photos require the owner, an operator, or a connected and unblocked relationship.
- Block lookup requires the current member or an operator.
- Matchmaking selects active and ready people.
- Application email cannot be changed to take over another identity.
- Operator email requests return a generic response.
- Production demo login is disabled by code.
- The Sentry test route returns 404 in production.

### Consent and decisions

- SMS sends require recorded SMS consent across introduction, resend, connection, feedback, group, and operator paths.
- STOP clears SMS consent, and START restores it.
- Operator quick-add requires explicit match confirmation and optional explicit SMS consent.
- Invite tokens expire after 14 days and reject closed match stages.
- Magic links are consumed atomically.
- Email and SMS inbound transient failures return 503 so providers can retry.
- Concurrent decisions are claimed atomically and covered by a database-backed race test.

### Media and uploads

- Invite photo URLs include the stored `.webp` extension.
- Failed upload processing removes the database row and local file when applicable.

### Accessibility and responsive behavior

- Public contrast failures were corrected.
- Header, footer, filters, labels, and tables have improved accessible names and relationships.
- The mobile drawer exposes dialog state, traps focus, closes on Escape, restores focus, and makes the background inert.
- Member and operator mobile layouts use the full viewport.
- Final home and apply accessibility scores are 100.

### Public truth and performance

- Unsupported automatic booking claims were removed.
- Invented testimonial attributions were removed.
- Privacy copy now describes passwordless tokens and optional SMS consent.
- The hero is server-rendered.
- Browser monitoring loads after the initial interaction window while still capturing early errors.
- Unused global toast code was removed.
- Font payload was reduced by loading one Fraunces style without unused axes.

### Release path

- The Docker image uses the Next standalone server.
- The runtime runs as the non-root `node` user.
- The Sentry upload token uses a BuildKit secret mount instead of a Docker argument or environment layer.
- GitHub Actions are pinned to full commit identifiers.
- README, deployment guidance, and the launch checklist now match the Fly and Neon architecture.

## Launch blockers

### Blocker 1: Shared photo storage

Current fallback storage writes to `/data/uploads`. Production has two Fly machines and one volume per machine. A photo written on one machine is not guaranteed to exist on the other.

Required proof:

- Configure shared object storage.
- Upload a real QA photo.
- Read it repeatedly through both Fly machines.
- Verify delete and replacement behavior.
- Remove or deliberately disable the local production fallback.

### Blocker 2: Durable introduction delivery

Database state and provider delivery are not one recoverable operation. A process exit can leave a match marked `connecting` or a sent timestamp without every message being accepted by its provider.

Required proof:

- Persist queued delivery work in the database.
- Record separate email and SMS statuses and provider message identifiers.
- Retry transient failures safely.
- Expose permanent failure to operators.
- Re-run after a forced process exit between the database write and provider call.

### Blocker 3: Concierge and booking truth

`concierge.say` writes an in-app message. No production schedule invokes `concierge:tick`, and confirmation does not send an external calendar invitation or create a reservation.

Required decision:

- Wire real delivery and booking operations, then test them end to end.
- Or disable the relevant operator actions and member-facing statements until that work exists.

### Blocker 4: Production demo cleanup

The branch disables demo login in production, but the live Fly app still lists demo-related secret names and contains a test-named operator.

Required proof:

- Remove the demo secret values from Fly.
- Audit or remove test operator records.
- Deploy the production-only guard.
- Confirm the demo login path cannot enumerate or authenticate operators.

## Other prelaunch requirements

- Enforce event capacity inside a transaction.
- Add database readiness to the health or release gate.
- Replace manual schema push with a reviewed, recoverable migration path.
- Give the watchdog working alert delivery and production Sentry access.
- Confirm Sentry source maps in a deployed build.
- Complete legal review of privacy, terms, SMS consent, retention, and safety language.
- Re-run the full authenticated browser suite after deployment.

## Release recommendation

Do not open public traffic from version 99. Resolve the four launch blockers, deploy this branch through the normal reviewed release path, and repeat the post-deploy checklist in `LAUNCH-CHECKLIST.md`.
