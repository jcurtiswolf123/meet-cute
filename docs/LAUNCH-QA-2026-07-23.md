## Executive finding

The initial launch QA found four product-integrity blockers. This branch resolves
all four in code and adds the release controls needed to verify them:

1. Photos now use Vercel Blob when configured and otherwise use Postgres. No
   production upload depends on a machine-local filesystem.
2. Email and SMS work now uses a durable database outbox with claim fencing,
   provider identifiers, bounded retries, stale-work recovery, authorization
   checks at send time, and operator-visible failures.
3. Unsupported automatic venue booking and calendar claims are removed from
   active product paths. Coordination is represented as manual.
4. Seeded test members and test-named operator access were removed from
   production. Code rejects production demo login regardless of environment
   secret values. The obsolete Fly secret names were removed.

The branch also adds transactional dinner capacity, schema-aware readiness,
checked-in Prisma migrations, a database release gate, production canaries, and
delivery monitoring.

## Safety and delivery changes

- SMS requires current consent when work is queued and again immediately before
  it is sent.
- Delivery is cancelled when a person opts out, blocks the other member, deletes
  an account, exits a match, or supersedes an invite token.
- Connection messages never disclose another member's phone number.
- Email delivery uses a provider idempotency key.
- Ambiguous SMS outcomes are not sent again automatically.
- One successful authorized connection channel per person is sufficient to
  finalize a connection.
- Stale workers cannot finalize or mutate a job claimed by a newer worker.
- Operator retries recheck current eligibility before requeueing work.

## Storage and database changes

- `PhotoAsset` stores shared photo bytes when Vercel Blob is not configured.
- Uploads enforce a final re-encoded size check and six-photo account quota.
- `DeliveryJob` stores durable per-channel work and claim state.
- `DeliveryJob.personId` cascades on account deletion.
- Dinner RSVP changes are serialized with a PostgreSQL advisory lock.
- `/readyz` confirms the exact Prisma models required by the new schema.
- Three checked-in migrations are applied in production, including the delivery
  safety migration.

## Release path

The GitHub workflow now performs:

1. Clean dependency installation and Prisma generation.
2. Type checking, lint, pure tests, and production build.
3. `prisma migrate deploy` against the production schema.
4. Database delivery, storage, capacity, and race tests.
5. A Fly deployment gated by machine readiness.
6. `/healthz`, `/readyz`, and home-page canaries.

The Docker image uses the Next standalone server as non-root user `node`.
`.dockerignore` excludes environment files, local QA artifacts, uploads, and
database dumps.

The production canary discovered one anonymous `/app` runtime error from a page
that assumed the layout redirect had already loaded a user. Next.js can render a
page concurrently with its layout. The hotfix added page-level authorization to
every protected member and operator page before any private query or user
dereference. Production browsing did not reproduce the error after version 106.
